# 护卫军成长体系 产品需求文档 (PRD)

---

## 模块一：执行摘要

### 1.1 业务背景与核心痛点
目前护卫军体系（网评员队伍）缺乏一套长效的荣誉激励与分层管理机制。现有的系统标签（尖兵/护卫队）属于静态的黑盒管理，导致以下核心痛点：
1. **荣誉感缺失**：C端网评员无法感知自身的成长路径，缺乏类似游戏“打怪升级”的荣誉正反馈。
2. **高净值用户流失预警迟缓**：缺乏动态的保级与降级机制，无法精准定位并激活处于“流失边缘”的核心战力。
3. **运营抓手单一**：作业下发与积分奖励过于同质化，无法根据网评员的阶梯分布精准调控“供需池”，导致低活跃用户无事可做，高活跃用户无高级作业可领。

### 1.2 产品解决方案概述
构建一套以 **“成长经验值 (XP)”** 为核心驱动力的**阶梯式成长体系**。
该体系将独立于现有的“积分消耗系统”，XP 仅代表荣誉资历，只获取不扣减（除违规与自然过期），通过明确的升级/降级/保级机制，牵引用户活跃。
同时，在 B端管理后台引入 **生态巡航智能体 (Eco-Cruiser AI)**，通过数据驱动的沙盘推演，辅助运营人员精准下发作业并调配资源。

### 1.3 成功衡量指标
*   **活跃指标**：体系上线后 3 个月内，L2 及以上段位的活跃留存率提升至 **85%** 以上。
*   **转化漏斗**：网评员“作业参与率”环比提升 **30%**，“高质量原创/转发作业的完成率”提升 **20%**。
*   **生态健康度**：保持成长生态大盘的“腰部及以上”核心力量占比稳定（例如 L2 占比稳定在 45%~50%，避免严重的“纺锤形”或“倒金字塔”分布失衡）。

## 模块二：C端体验 (用户体验与功能)

### 2.1 用户画像
*   **新进网评员 (L1 新秀)**：刚加入体系或历史极少参与任务的用户。他们需要明确的上手指导、低门槛的破冰任务（如点赞/评论），以及对下个段位特权的强烈渴望。
*   **主力网评员 (L2 熟练 / L3 专家)**：体系的中坚力量。对积分和特权有刚性需求。他们最关心“还有多少经验升级”以及“我的高分会不会过期导致降级”。
*   **核心意见领袖 (L4 大师)**：极高净值用户。需要极强的专属荣誉感（发光徽章、特殊头衔、线下实物勋章等）和防跌落的“免死金牌”（长效保级期与软着陆保护）。

### 2.2 核心功能模块设计

#### 2.2.1 个人中心与资产卡片区
*   **功能描述**：作为 C 端用户的门面，直观展示荣誉与资产。
*   **展示字段**：
    *   **当前段位徽章**：L1 ~ L4 专属图标（高段位带动态流光特效）。
    *   **历史最高荣誉印记**：若用户曾达到高段位但降级，需以底纹或小印章形式展示（如“历史最高：L4大师”），强化终身荣誉感。
    *   **积分与 XP 隔离展示**：明确切割“可消费积分”与“成长经验值(XP)”，点击各带独立明细入口。

#### 2.2.2 动态进度条与保级预警区 (核心体验区)
*   **功能描述**：化解用户的进度焦虑与“不明掉分”的客诉风险。
*   **动态展示逻辑**：
    *   **常态**：显示当前 XP 及距离下一等级的进度条（如：`距离 L3 专家还差 250 XP`）。
    *   **效期流水预警**：常态露出“本月新增：+X XP | 下月底即将过期：-Y XP”。
    *   **保级状态高亮**：若用户处于 90 天保级期内，区域标红高亮：“🛡️ 季度保级保护生效中，剩余 [N] 天”。
    *   **软着陆提示**：若用户触发退级但受老兵保护，提示：“🛡️ 触发老兵保护，本次免于掉落至 [低一段位]”。

#### 2.2.3 任务大厅与特权展示区
*   **功能描述**：引导用户向目标段位攀爬的转化路径。
*   **特权轮播图**：横向滑动展示所有段位权益（如 优先领单权）。高于当前段位的权益置灰加锁，刺激升级。
*   **快捷赚分入口**：点击直达“任务大厅”，按用户当前段位权限，智能置顶可接的最高 XP 奖励任务。

### 2.3 用户故事与验收标准
*   **场景 1**：作为一名 **L3 专家网评员**，我希望能提前知道我哪些 XP 快过期了，以便我能及时去做任务补齐，防止掉段。
    *   *验收标准 1*：系统必须在 XP 过期前 30 天，在进度条区域显眼位置提示即将过期的 XP 数量。
    *   *验收标准 2*：点击预警提示，可下钻查看具体的过期明细账单。
*   **场景 2**：作为一名 **曾经的大师网评员**，我因为休假几个月没有做任务导致降级了，我希望我的历史贡献不要被完全抹杀。
    *   *验收标准 1*：即使当前为 L2，个人卡片页依然会保留“历史最高：大师”的水印。
    *   *验收标准 2*：首次降级时，必须触发“软着陆”机制，只允许降一级，不能直接从 L4 掉到 L1。
*   **场景 3**：作为一名 **新网评员**，我想要快速弄懂这套复杂的体系。
    *   *验收标准 1*：首次登录必须弹出全屏的《用户指南》引导弹窗。
    *   *验收标准 2*：所有带名词解释的地方（如保级期、过期时间）必须支持点击问号弹窗解释。

## 模块三：B端管理 (后台业务管理)

本模块作为整个成长体系的“中枢神经”，供业务侧运营人员进行日常的健康度巡航、规则配置与定向干预。

### 3.1 核心后台系统模块

#### 3.1.1 B端大盘健康度监控 (大盘数据与XP资产流水)
*   **全局联动与组织下钻**：采用“左侧组织树 + 右侧大盘”结构。右侧所有数据（折线图、漏斗、流水）均必须严格根据左侧树勾选的**物理架构节点（至多下钻至党支部）**进行联动过滤。
*   **同级横向对比算法 (木桶短板下钻)**：为了防止不同分支的组织结构嵌套深度不一致（如某些分公司没有三级党委），在生成“段位分布对比图”时，系统必须自动探寻所有选中节点的“最低公共业务层级”作为 X 轴，实现真正的平级 PK。
*   **全景资产对账表**：底部提供全员 XP 流转的对账流水（正向收入绿、自然扣除灰、违规扣除红、人工调账黄）。

#### 3.1.2 成长规则配置 (引擎中枢)
*   **段位与特权门槛表**：维护各级（L1~L4）段位名称、晋升的基准 XP 门槛。
*   **全局引擎规则**：控制体系的全局时间参数，包括 365天滚动失效周期、高段位专属保级期时长（如 90 天），以及软着陆“防跌落”机制的总开关。
*   **作业 XP 映射池**：配置各类“作业”（基础互动、原创、转发）对应的单次获得 XP 产出值，作为定价基准基石。

#### 3.1.3 护卫军作业任务配置 (作业管理)
*   **功能定位**：具体的护卫军作业任务实例的发布端。
*   **功能描述**：运营创建任务时，除了设置任务本身的要求和积分悬赏外，还可勾选该任务所属的“成长标签分类”，系统会自动从“作业 XP 映射池”中提取当前任务的默认 XP 奖励并一并挂载至工单下发。

#### 3.1.4 反馈审核列表 (审核管理)
*   **功能定位**：针对高难度作业（如原创发文），网评员提交证明后的 B 端审核端。
*   **功能描述**：审核人员查看用户提交的截图/链接。审核通过后，系统除了向网评员发放可消费“积分”外，还要**原子性（Atomic）地**将绑定的 “XP” 发放至用户的成长账户中，并记入流水。

#### 3.1.5 用户管理与干预 & 文章管理
*   **个体干预与调账**：用于特殊处理客诉。支持对单个网评员进行 XP 手动补发、违规恶意扣除甚至直接锁定/解锁当前段位（无视算法直接提权）。
*   **文章精选维护**：对接系统内的阅读文章分发，支持配置诸如“阅读满 X 篇奖励 Y 个 XP”或“王牌护卫军专属阅读库”等长尾赋能机制。

### 3.2 用户故事与验收标准
*   **场景 1**：作为 **大盘运营人员**，我希望能直观地看出哪些分公司的 L3 段位掉级最严重，以便我针对性地下发高分作业去捞他们。
    *   *验收标准 1*：大盘必须提供“保级失败降级人次”的负面指标统计，并在组织柱状图中用红色突显下行流失数据。
    *   *验收标准 2*：左侧组织树必须支持模糊搜索。
*   **场景 2**：作为 **配置管理员**，由于近期预算问题，我需要把点赞这类低门槛动作的 XP 产出从 5 降到 2，但我不想影响已经发出去的作业。
    *   *验收标准 1*：XP 映射池的修改仅对配置发布后**新生成**的作业实例生效，严禁溯及历史未结算作业。
*   **场景 3**：作为 **审核专员**，我审核通过一条高质量反馈后，希望网评员能同时收到积分和XP，而不是我操作两遍。
    *   *验收标准 1*：审核通过动作将触发分布式事务，确保积分到账和 XP 到账在同一次流水回执中完成。

## 模块四：核心状态机与资产规则

本模块是整个体系的业务算法核心，用于规范 XP 在系统内的流转生命周期，以及段位的晋升与降级判定逻辑。

### 4.1 阶梯段位与经验值门槛
系统统一采用 **1 积分 = 1 XP** 的基准兑换比例（仅限赚取时等比累加），建立如下 12 个月滚动生命周期段位：

| 段位名称 | 经验值 (XP) 门槛 | 核心业务特权 | 尊享荣誉特权 |
| :--- | :--- | :--- | :--- |
| 🌱 **L1 (新秀护卫军)** | `0 - 199 XP` | 基础作业领取权 | 无 |
| ⚙️ **L2 (熟练护卫军)** | `200 - 999 XP` | 常规内容作业领取权 | 基础护卫军发光标识 |
| 🎖️ **L3 (专家护卫军)** | `1000 - 2499 XP` | **解锁重磅核心任务**<br>高优任务优先领取权 | 专家专属身份标识<br>专属客服响应通道 |
| 👑 **L4 (大师护卫军)** | `≥ 2500 XP` | **含专家全部特权**<br>大师专属身份标识<br>直面高管圆桌会 |

### 4.2 XP 流水账单与增减类型
XP（成长经验值）是一套典型的**单向累加、到期清退**模型。任何形式的商城积分兑换，**绝对不允许消耗 XP**。
底层流水的“业务类型”严格限定为以下 4 种枚举值：

1.  **作业产出 (+)**：
    *   最核心的正向来源。用户完成各类护卫军作业（如原创、评论、转发），审核通过后系统自动下发的 XP 奖励。
2.  **人工调账 (±)**：
    *   包含上线初期的“历史老兵标签平移”空投发放，以及日常应对客诉时，由 B 端运营在后台手动触发的增补或扣回。
3.  **到期扣除 (-)**：
    *   **365天滚动失效机制**（自然核销）：用户获得的每一笔正向 XP 都带有时间戳，满 365 天后，由系统夜间定时跑批任务自动触发扣除。
4.  **违规扣减 (-)**：
    *   针对恶意刷单、提交违规反馈等行为，系统自动或人工触发的惩罚性扣减。此扣减不受 365 天逻辑限制，即刻扣除。

### 4.3 降级保护状态机
考虑到高阶用户的培养成本极高，直接因为 XP 过期导致断崖式降级会引发严重客诉，系统必须引入“保级缓冲”与“软着陆”状态机。

#### 4.3.1 定级状态定义
根据 B 端用户的生命周期与 XP 水位，系统将用户划分为以下 4 种定级状态：
1.  **正常 (XP达标)**：用户的实时总 XP ≥ 当前所在段位的基准门槛。属于最健康的活跃状态。
2.  **保级缓冲期**：用户的实时总 XP < 所在段位门槛，但得益于高段位（L3/L4）专属特权，正处于 90 天保级倒计时内。此时**段位不降，特权保留**，但大盘健康度会将其视为风险用户。
3.  **荣誉保底**：保级期满后 XP 仍未达标，本应发生跨级断崖式降级（例如本应从 L4 直接掉到 L1）。但触发了“老兵软着陆防跌落”机制，被系统强行托底在 L3。此时用户的状态即为“荣誉保底”。
4.  **退出暂不评级**：极度长期的沉睡用户，XP 被 365天规则彻底扣光，且不满足任何保底条件，掉出核心护卫军评级体系。

#### 4.3.2 核心状态流转规则
*   **正常 ➞ 保级缓冲期**：因 XP 滚动过期或违规扣减，导致实时 XP 跌破门槛，且该段位配置了保级期特权（L3/L4）时触发。
*   **保级缓冲期 ➞ 正常**：在倒计时结束前，用户通过做作业重新赚取 XP，使 `实时XP ≥ 门槛`，成功自救。
*   **保级缓冲期 ➞ 荣誉保底**：倒计时结束仍未达标，但满足软着陆规则，系统只降一级并打上“荣誉保底”标记。
*   **任何状态 ➞ 退出暂不评级**：用户长期无任何作为，XP 归零。

### 4.4 用户故事与验收标准
*   **场景 1**：作为 **后台跑批程序**，我每天凌晨需要处理几十万条到期的 XP 记录，并计算所有人的状态。
    *   *验收标准 1*：必须确保事务的一致性，先生成“过期扣除流水”，再计算用户的总 XP，最后触发状态机判定。
    *   *验收标准 2*：状态判定必须依赖于底层配置表的 `保级期时长(天)` 参数，禁止硬编码为 90 天。

## 模块五：生态巡航智能体

本模块定义了 B 端“生态巡航智能体”的业务逻辑与交互标准。该智能体通过调用大语言模型（LLM）与内部数据仓库的 API，为运营人员提供“诊断-归因-开方-推演”的闭环决策支持。

### 5.1 触发机制与工作流

智能体的介入并非时刻开启，而是由大盘健康度数据驱动：
1.  **被动触发（大盘异动报警）**：当系统监控到某些核心指标（如“连续 7 天 L3 段位活跃率跌破 30%”）触发阈值时，自动拉起智能体生成诊断报告并推送给运营。
2.  **主动问诊（人工巡航）**：运营人员在 B 端大盘页面主动点击“生成智能诊断”。系统将当前所选组织树节点下的**大盘聚合脱敏数据**打包发送给大模型。

### 5.2 核心诊断四大件

#### 5.2.1 核心预警提取
AI 根据输入的时序数据，找出波动最剧烈或偏离安全基准线最远的一项核心指标，提炼为一句话预警（如：“L3 专家群体的『优质长文原创率』较上月骤降 45%”）。

#### 5.2.2 专家级 AI 归因
不仅要说出“跌了”，更要通过交叉对比找出“为什么跌”。
例如：结合“作业下发频率”与“XP收益表”，AI 分析出“本月下发的高难度长文作业占比极低，导致 L3 专家无高分可赚，丧失登录动力”。

#### 5.2.3 可落地运营处方
杜绝宏观套话（如“建议加强运营”），AI 必须结合**护卫军的具体功能机制**给出具体的开方建议。
例如：“建议针对全省 L3/L4 标签用户，定推 3 个单次价值高达 300 XP 的『王牌破局』定向长文作业。”

#### 5.2.4 沙盘推演预测
基于给定的处方，结合历史转化率大盘数据，预测该方案执行后可能带来的涨幅收益（如“预计推行后 7 天内，L3 活跃率可回调至 42% 左右”）。

### 5.3 智能体输出数据结构与 Prompt 预设

为了保证大语言模型输出的内容能被前端原型图完美渲染，并能直接转化为系统的参数下发，智能体的输出**必须强制遵循以下 JSON 标准结构**：

```json
{
  "alert": {
    "title": "预警标签 (如：腰部流失预警)",
    "description": "具体的异动指标描述文本"
  },
  "attribution": "专家级归因文本，说明导致异动的交叉因素",
  "prescription": "可落地的具体运营处方文本",
  "simulation_result": "预计提升的指标预测（如：活跃率回升至 45%）",
  "recommended_action": {
    "action_type": "任务下发 | 规则调整",
    "target_level": "L3",
    "task_type": "原创 / 转发 / 评论",
    "xp_reward_suggestion": 200
  }
}
```

### 5.4 用户故事与验收标准
*   **场景 1**：作为 **大盘运营人员**，我希望智能体不仅告诉我该怎么办，还能帮我把“怎么办”填到派发系统的表单里。
    *   *验收标准 1*：智能体输出的 JSON 中必须包含 `recommended_action` 结构化对象。
    *   *验收标准 2*：在 B 端交互上，运营点击“采纳建议”按钮后，系统需自动跳转到“作业派发”页面，并将目标人群、任务类型和建议的 XP 自动填入表单。

## 模块六：老系统过渡与数据迁移

作为一套全新的成长体系，我们必须极其慎重地处理与老系统的兼容过渡，尤其是对那些曾经立下汗马功劳的“历史老兵”的激励延续。

### 6.1 历史老兵数据平移策略

老系统虽然只有“尖兵/护卫队”这样的静态标签，但它承载了大量核心用户的荣誉感。
因此，在新旧体系切换时，**绝对不能搞“一刀切、全员从零开始”**。

*   **空投直升机制**：
    *   在系统上线前夜，通过后台跑批，对当前拥有“尖兵”标签的用户，**直接空投发放等同于 L4 大师的基准 XP 值**（建议 2500 XP）。
    *   对当前拥有“护卫队”核心标签的用户，直接空投发放等同于 L3 专家的基准 XP 值（建议 1000 XP）。
*   **老兵保级豁免**：
    *   这些因“空投”直接获得高段位的历史老兵，其状态默认进入 `Normal` 且立刻享有 90 天的新手保护期（保级倒计时延后 90 天启动），确保他们有充足的时间适应新版的派单规则。

### 6.2 系统灰度上线方案

为了防止 XP 定价体系崩溃或通货膨胀，建议采用两阶段发布：
*   **Phase 1 - 白名单试运行 (Alpha)**：挑选 2 个分公司所在省份作为白名单，灰度开放 C 端个人中心。验证 2 周内的“作业下发频率”与“XP产出膨胀率”是否匹配我们在 B 端的沙盘测算模型。
*   **Phase 2 - 全量开放 (GA)**：全员开放，同时 B 端正式激活“生态巡航智能体”，开始每日对大盘进行自动化问诊。

### 6.3 新老规则兼容说明

*   **积分消耗系统**：老体系中的“消费积分”保留且继续沿用，商城兑换逻辑不受任何影响。
*   **双轨并行**：在白名单试运行期间，老标签（尖兵）与新段位（L4）会在系统中双轨并行存储。待 GA 上线 3 个月后，通过公告正式废弃老的静态标签查询，全面转为动态 XP 体系。
